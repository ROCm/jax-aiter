# JAX-AITER Design Overview

This document explains how `jax-aiter` is structured, how requests flow from JAX to ROCm kernels, and where to extend the system safely.

## 1) Purpose and scope

`jax-aiter` integrates AMD AITER operator kernels into JAX using XLA FFI:

- Python/JAX APIs are exposed in `jax_aiter/*`.
- Native FFI bridge handlers live in `csrc/ffi/*`.
- Kernel implementations and code objects (`.co`) are provided by AITER in `third_party/aiter`.

The repository is designed so users call normal JAX functions while performance-critical kernels execute in ROCm native code.

## 2) High-level architecture

```text
User JAX code
  -> jax_aiter Python API (custom_vjp wrappers + shape/dtype checks)
     -> jax.ffi.ffi_call("SymbolName")
        -> FFI registry (loads .so modules, resolves symbol, registers target)
           -> csrc/ffi/* bridge function (XLA_FFI_DEFINE_HANDLER_SYMBOL)
              -> AITER runtime entrypoint / ASM kernel launcher
                 -> HIP kernel execution on GPU
```

## 3) Main layers

### 3.1 Python API layer (`jax_aiter/*`)

Key operator modules:

- `jax_aiter/mha/mha.py`
- `jax_aiter/rmsnorm/rmsnorm.py`
- `jax_aiter/gemm/gemm.py`
- `jax_aiter/gemm_fp8/gemm_fp8_mi350.py`
- `jax_aiter/gemm_i8/gemm_i8.py`
- `jax_aiter/gemm_fp4/gemm_fp4.py`
- `jax_aiter/flatmm_fp8/flatmm_fp8.py`

Common patterns used by these modules:

1. Ensure the FFI symbol is registered (`register_ffi_target(...)`).
2. Build a `jax.ffi.ffi_call(...)` with explicit output shapes/dtypes.
3. Wrap with `jax.jit` for compilation/caching.
4. Use `jax.custom_vjp` where custom backward behavior is needed.

Examples:

- MHA: forward/backward both call unified FFI handlers and apply route guards for stable execution modes.
- RMSNorm: forward uses AITER CK kernel, backward is computed in JAX.
- BF16 GEMM: forward and backward both use FFI (backward is expressed with transposed GEMM calls).

### 3.2 Runtime/compat layer (`jax_aiter/ja_compat`, `jax_aiter/ffi`)

- `jax_aiter/__init__.py` calls `set_aiter_asm_dir()` at import time.
- `jax_aiter/ja_compat/config.py` decides library and kernel paths for:
  - development mode (`JA_ROOT_DIR`)
  - installed package mode (`jax_aiter/_lib`, `jax_aiter/_hsa`)
- `jax_aiter/ffi/registry.py` owns dynamic loading and registration:
  - loads umbrella lib `libjax_aiter.so`
  - loads module libs from `build/{aiter_build,jax_aiter_build}` (or packaged `_lib`)
  - resolves symbol pointers and registers them into JAX FFI

### 3.3 Native bridge layer (`csrc/ffi/*`)

Each bridge defines an exported FFI symbol via `XLA_FFI_DEFINE_HANDLER_SYMBOL` and translates XLA buffers/attrs into AITER kernel args.

Current wired symbols:

- `MhaFwdUnifiedJA`
- `MhaBwdUnifiedJA`
- `RmsnormFwdJA`
- `GemmFwdJA`
- `GemmFp8Mi350FwdJA`
- `GemmI8FwdJA`
- `GemmFp4FwdJA`
- `FlatmmFp8FwdJA`

Bridge responsibilities include:

- validating shapes/dtypes/constraints
- selecting kernel/config variants
- preparing argument structs and temporary buffers
- launching kernels on HIP streams

### 3.4 AITER kernel/codegen layer

- JIT modules: built from AITER using `jax_aiter/jit/build_jit.py`
- FFI modules: built by Makefile targets from `csrc/ffi/*`
- ASM config headers: auto-generated by AITER `hsa/codegen.py` for GEMM variants
- Kernel binaries (`.co`): consumed through `AITER_ASM_DIR` path

## 4) Build and packaging design

Build flow:

1. `make` builds umbrella library (`build/jax_aiter_build/libjax_aiter.so`).
2. `python3 jax_aiter/jit/build_jit.py` builds AITER JIT modules to `build/aiter_build`.
3. `make ja_mods` builds jax_aiter FFI handler modules (`*_ja.so`).
4. `pip install .` packages Python code and copies:
   - `build/*/*.so` -> `jax_aiter/_lib/**`
   - `third_party/aiter/hsa/**` -> `jax_aiter/_hsa/**`

The wheel is explicitly marked non-pure because it ships platform-specific `.so` artifacts.

## 5) Data/control flow examples

### 5.1 Flash Attention

- `flash_attn_func(...)`
  -> `_flash_attn_forward(...)`
  -> `mha_fwd_unified(...)`
  -> FFI call `MhaFwdUnifiedJA`
  -> bridge calls `aiter::mha_fwd(...)`

Backward:

- custom VJP backward
  -> `mha_bwd_unified(...)`
  -> FFI call `MhaBwdUnifiedJA`
  -> bridge calls `aiter::mha_bwd(...)`

### 5.2 BF16 GEMM

- `gemm(a, b)` where output is `a @ b.T`
  -> FFI call `GemmFwdJA`
  -> bridge selects ASM kernel heuristically
  -> launches via `AiterAsmKernel`

Backward uses the same forward bridge with transformed inputs (`dOut`, `B.T`, `A.T`) to compute `dA` and `dB`.

## 6) Testing strategy

Tests under `tests/` validate:

- numerical accuracy vs JAX reference implementations
- shape/dtype contracts
- backward correctness for custom VJPs
- architecture-specific behavior and skip rules (gfx942 vs gfx950)
- smoke coverage for major operator families

Notable test focus:

- large MHA matrix of cases including varlen/MQA/GQA/causal/dropout regressions
- RMSNorm forward and JAX-based backward parity
- BF16 GEMM forward and backward parity

## 7) Extending the system (new op checklist)

To add a new operator:

1. **Native bridge**: add `csrc/ffi/<op>/<op>_ja.cu` with a unique FFI symbol.
2. **Build wiring**: add the module target to `Makefile` (`ja_mods` path).
3. **Registry wiring**: add symbol -> module mapping in `jax_aiter/ffi/registry.py`.
4. **Python API**: create `jax_aiter/<op>/<op>.py` wrapper (`ffi_call`, optional `custom_vjp`).
5. **Package exports**: update package `__init__.py` exports as needed.
6. **Tests**: add forward/backward/shape tests and arch-specific skip logic where required.
7. **Docs**: update this design overview and top-level README supported-op list.

## 8) Current implementation notes

- The design intentionally keeps high-level APIs JAX-native while isolating GPU-specific concerns inside FFI bridges.
- Architecture-sensitive behavior is centralized around `GPU_ARCHS` detection and `.co` availability.
- In this branch, `gemm_a8w8` and `wv_splitkq` wrappers exist in Python but their FFI symbols are not part of the current registry symbol map / default native build targets yet.
